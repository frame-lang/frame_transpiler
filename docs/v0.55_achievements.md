## Key Fixes

### 1. Indexed Function Call Argument Handling ✅
**Problem**: When calling functions through list indexing like `operations[0](10, 5)`, arguments were incorrectly wrapped in an extra tuple, generating `operations[0]((10, 5))`.

**Solution**: Modified the parser to use `expr_list()` for parsing indexed call arguments, preventing incorrect tuple creation while preserving the correct parsing context.

**Impact**: 
- Function references stored in collections now work correctly
- Higher-order function patterns fully supported
- No regression in tuple literal handling

### 2. Tuple Literal Preservation ✅
**Problem**: Tuple literals passed as function arguments were being incorrectly unwrapped, turning `func((1, 2, 3))` into `func(1, 2, 3)`.

**Solution**: Removed overly aggressive tuple unwrapping logic from the visitor, relying on the parser's correct handling of different contexts.

**Impact**:
- Pattern matching with tuple patterns works correctly
- Deliberate tuple arguments preserved as intended
- Match-case statements fully functional

## Technical Implementation

### Parser Enhancement
The key insight was that the parser needed to distinguish between different contexts when parsing parenthesized expressions:

```rust
// For indexed calls, use expr_list() to prevent tuple creation
let args = match self.expr_list() {
    Ok(Some(ExprListT { expr_list_node })) => expr_list_node.exprs_t,
    Ok(Some(_)) => return Err(ParseError::new("Invalid expression list in indexed call")),
    Ok(None) => Vec::new(),
    Err(e) => return Err(e),
};
```

### Context-Aware Parsing
The `expr_list()` function sets the `is_parsing_collection` flag to prevent comma-separated values from being incorrectly parsed as tuples in function call contexts.
