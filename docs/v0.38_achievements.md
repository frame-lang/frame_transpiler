## Overview

Frame v0.38 delivers four major improvements: complete collection syntax support, full first-class function support, lambda expression support, and alignment with Python's logical operator syntax. These changes provide comprehensive Python-style collection operations, complete functional programming capabilities including both named functions and lambdas as first-class values, and remove C-style operators in favor of Python's more readable keyword-based approach.

## Major Achievement 1: Complete Collection Syntax Support

### All 8 Collection Patterns Now Supported

Frame v0.38 implements comprehensive collection syntax with both literal and constructor forms for all Python collection types:

#### Set Collections
```frame
// Set constructor with arguments
var s = set(1, 2, 3)           // → {1, 2, 3}

// Set literal syntax
var s_set = {1, 2, 3}          // → {1, 2, 3}
```

#### List Collections
```frame
// List constructor with arguments  
var l = list(1, 2, 3)          // → [1, 2, 3]

// List literal syntax
var l2 = [1, 2, 3]             // → [1, 2, 3]
```

#### Dictionary Collections
```frame
// Dict constructor (Python-compliant)
var d = dict()                 // → {}

// Dictionary literal syntax
var d2 = {"a": 1, "b": 2}      // → {"a": 1, "b": 2}
```

#### Tuple Collections
```frame
// Tuple constructor with arguments
var t = tuple(10, 20, 30)      // → (10, 20, 30)

// Tuple literal syntax
var t2 = (10, 20, 30)          // → (10, 20, 30)
```

### Implementation Details

#### Parser Enhancements (parser.rs)
- **Set/Dict Disambiguation**: Added `dict_or_set_literal()` function to distinguish `{}` syntax based on colon presence
- **Tuple Detection**: Added `expr_list_or_tuple()` to detect tuple vs parenthesized expression patterns
- **AST Integration**: New `SetLiteralNode` and `TupleLiteralNode` with proper visitor support

#### Python Visitor Transformations (python_visitor.rs)
- **Collection Constructors**: Special handling in `visit_call_expression_node` for `list()`, `set()`, `tuple()`, `dict()`
- **Literal Generation**: Constructor calls transformed into Python literals for optimal code generation
- **Python Compliance**: `dict()` kept as-is for proper Python syntax compatibility

#### Key Technical Decisions
- **Empty `{}` defaults to dictionary** for Python compatibility
- **Single-element tuples** automatically get trailing comma in Python output
- **Constructor transformation** converts `list(1,2,3)` → `[1,2,3]` for cleaner generated code

## Major Achievement 3: First-Class Function Support (NEW)

### Complete First-Class Functions
Frame v0.38 now supports both named functions and lambdas as first-class values:

```frame
// Functions as values
fn add(a, b) {
    return a + b
}

fn multiply(a, b) {
    return a * b
}

// Assign function to variable
var my_func = add
var result = my_func(3, 4)  // 7

// Pass function as parameter
fn apply_op(op, x, y) {
    return op(x, y)
}

result = apply_op(multiply, 5, 3)  // 15

// Return function from function
fn get_operation(name) {
    if name == "add" {
        return add
    } else {
        return multiply
    }
}

var op = get_operation("add")
result = op(10, 20)  // 30

// Store functions in collections
var operations = [add, multiply]
var ops_dict = {"plus": add, "times": multiply}

result = operations[0](2, 3)  // 5
result = ops_dict["times"](4, 5)  // 20
```

### Implementation Details

#### AST Enhancement (ast.rs)
- Added `FunctionRefT { name: String }` variant to `ExprType` enum
- Updated visitor trait with `visit_function_ref_node()` methods

#### Parser Changes (parser.rs)
- Modified `call()` function to detect function references (identifiers without parentheses)
- Added `SymbolType::FunctionScope` handling in `get_identifier_scope()`
- Two-pass strategy: functions recognized as references in second pass

#### Python Visitor (python_visitor.rs)
- Implemented `visit_function_ref_node()` - generates clean function name
- No special syntax needed - Python naturally supports first-class functions

## Major Achievement 4: Python-Only Logical Operators

### Breaking Changes Implemented
- **Removed C-style operators**: `&&`, `||`, `!` now generate compilation errors
- **Python keywords required**: Only `and`, `or`, `not` accepted
- **Clear migration path**: Scanner provides helpful error messages

### Implementation Details

#### Scanner Changes (scanner.rs)
```rust
// Old operators now generate errors with migration guidance
'&' => {
    if self.match_char('&') {
        self.error(self.line, "Operator '&&' has been removed. Use 'and' keyword instead.");
    }
}
'|' => {
    if self.match_char('|') {
        self.error(self.line, "Operator '||' has been removed. Use 'or' keyword instead.");
    }
}
'!' => {
    if self.peek() != '=' {
        self.error(self.line, "Operator '!' has been removed. Use 'not' keyword instead.");
    }
}
```

#### Parser Changes (parser.rs)
- Removed `TokenType::LogicalAnd`, `TokenType::PipePipe`, `TokenType::Bang` from match statements
- Only accepts `TokenType::And`, `TokenType::Or`, `TokenType::Not`

#### AST Changes (ast.rs)
- Removed mappings for old C-style token types
- Simplified operator type conversions

## Post-Release Fixes (2025-09-07)

### Lambda in Return Statements
**Issue**: Return statements couldn't parse lambda expressions  
**Fix**: Changed return statement parsing to use `expression()` instead of `equality()`, allowing full expression parsing including lambdas  
**Impact**: Functions can now return lambda expressions directly: `return lambda x: x + n`

### Domain Block Dictionary Initialization
**Issue**: Domain blocks with dictionary literals caused parse errors when appearing before other system blocks  
**Fix**: Identified parser limitation requiring domain blocks to appear last in system definitions  
**Workaround**: Move domain blocks to the end of system definitions when using dictionary literals  
**Long-term**: Requires parser refactoring to allow flexible block ordering

### Array Indexing with Function Calls (Fixed 2025-09-07)
**Issue**: Parser couldn't handle calling functions stored in arrays/dicts directly: `operations[0](args)`  
**Solution**: Added synthetic `@indexed_call` AST node to represent indexed function calls  
**Implementation**: 
- Parser detects `(` after array/dict indexing and creates special call node
- Python visitor handles `@indexed_call` nodes without adding dot separator
- Supports nested patterns: `matrix[0][1](x, y)`
**Impact**: Enables dispatch tables, function arrays, and dynamic function selection patterns

### Return Syntax Standardization (2025-09-07)
**Breaking Change**: Removed `return = value` syntax  
**Rationale**: Clearer distinction between regular returns and interface return values  
**Migration**: All uses of `return = value` must be changed to `system.return = value`  
**Error Message**: Parser now provides helpful error: "Use 'system.return = value' to set the interface method return value"  
**Impact**: Updated 35 test files to use the standardized syntax

### Updated Test Results
After all fixes, Frame v0.38 achieves **97.9% test success rate** (284/290 tests passing).
