## 🎯 Major TypeScript Visitor Improvements

### Critical Fixes for TypeScript Code Generation
Successfully resolved multiple TypeScript visitor issues including debug output pollution and system parameter support.

## 🐛 Bugs Fixed

### 1. Debug Output Pollution
**Problem:** Unconditional debug statements were outputting to stderr even in production builds, causing test failures.

**Solution:** Added proper environment variable gating for all debug statements:
- Debug output now only appears when `FRAME_TRANSPILER_DEBUG=1` is set
- Clean output in production builds
- No more test failures due to debug pollution

### 2. System Parameter Support
**Problem:** TypeScript visitor was generating constructors with no parameters, even when systems had parameters defined.

**Impact:** 
- Systems with start state parameters failed to compile
- Systems with enter event parameters failed to compile  
- Systems with domain parameters failed to compile

**Solution:** Complete implementation of system parameter support:
- Constructor now accepts all system parameters (state, enter, domain)
- Proper parameter passing to FrameCompartment initialization
- Correct state args and enter args objects creation
- Domain variables initialized from domain parameters when provided

**Example of fixed output:**
```typescript
// Before (incorrect):
constructor() {
    this._compartment = new FrameCompartment('__state_S1');
}

// After (correct):
constructor(p1: any) {
    this._compartment = new FrameCompartment('__state_S1', null, null, {'p1': p1}, null);
}
```

### 3. FrameCompartment Parameter Order Fix
**Problem:** State args and enter args were being passed in wrong positions to FrameCompartment constructor.

**Solution:** Fixed parameter order to match FrameCompartment signature:
- Correct order: `state, enterArgs, exitArgs, stateArgs, stateVars`
- Was incorrectly: `state, stateArgs, null, enterArgs, null`
