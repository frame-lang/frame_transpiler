# Frame v0.62 Progress - Call Chain Refactoring

**Start Date:** September 20, 2025  
**Status:** 🚧 IN PROGRESS  
**Goal:** Refactor call chain processing to move semantic resolution from visitor to parser

## Overview

Frame v0.62 addresses the architectural debt in call chain processing by moving semantic analysis from the visitor layer (where it causes 350+ lines of complex duplication) to the parser layer (where it belongs). This refactoring will significantly simplify code generation and improve maintainability.

## Architecture Changes

### Old Architecture (Current)
```
Parser → AST (minimal context) → Visitor (complex resolution + generation)
```

### New Architecture (v0.62)
```
Parser → Semantic Analyzer → AST (with resolved types) → Visitor (simple generation)
```

## Key Components Added

### 1. `ResolvedCallType` Enum
```rust
pub enum ResolvedCallType {
    Action(String),              // Internal action call
    Operation(String),           // Internal operation call  
    SystemOperation {            // Qualified system operation
        system: String,
        operation: String,
    },
    ModuleFunction {             // Qualified module function
        module: String,
        function: String,
    },
    External(String),            // True external function
}
```

### 2. `SemanticAnalyzer` Module
- Resolves call expressions to their semantic type
- Uses symbol table for accurate resolution
- Runs during parser's second pass

### 3. Feature Flag
- `enable_semantic_resolution`: Controls new behavior
- Default: `false` (uses old behavior)
- Allows gradual migration and testing

## Benefits When Complete

1. **Code Reduction**: ~350 lines → ~50 lines in visitor
2. **Single Responsibility**: Parser analyzes, visitor generates
3. **Performance**: Semantic resolution happens once
4. **Maintainability**: Clear separation of concerns
5. **Bug Prevention**: Fixes double qualification naturally
