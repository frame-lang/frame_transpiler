# Frame v0.64 Plan - Visitor Simplification Using Resolved Types

**Target Date**: 2025-09-21  
**Branch**: v0.30  
**Goal**: Use semantic resolution to simplify Python visitor code generation

## Overview

Now that v0.63 provides accurate semantic resolution (Actions, Operations, External calls correctly identified), v0.64 will leverage this information to simplify the complex call chain logic in the Python visitor.

## Current Problem

The Python visitor contains 350+ lines of complex logic to determine:
- Is this an action call that needs underscore prefix?
- Is this an operation that needs self?
- Is this a static method call?
- Is this an external function?

This logic is duplicated and complex, making the visitor hard to maintain.

## Solution: Use Resolved Types

With v0.63's accurate resolution, the visitor can simply check `call_expr.resolved_type`:

```rust
// Instead of complex analysis...
match &call_expr.resolved_type {
    Some(ResolvedCallType::Action(name)) => {
        // Generate action call: self._name()
    }
    Some(ResolvedCallType::Operation(name)) => {
        // Generate operation call: self.name()
    }
    Some(ResolvedCallType::External(name)) => {
        // Generate external call: name()
    }
    // ... other cases
    None => {
        // Fallback to current logic if no resolution
    }
}
```

## Implementation Plan

### Phase 1: Add Feature Flag
- Add `use_semantic_resolution` flag to PythonConfig
- Allow gradual migration of visitor code
- Maintain backward compatibility

### Phase 2: Simplify Call Generation
- Replace complex call chain analysis with resolved type checks
- Remove duplicate logic for determining call types
- Simplify `visit_call_expression_node()` method

### Phase 3: Clean Up
- Remove redundant helper methods
- Delete complex call chain analysis code
- Update tests to verify behavior unchanged

## Benefits

1. **Code Reduction**: Remove ~350 lines of complex logic
2. **Maintainability**: Single source of truth for call types
3. **Performance**: No redundant analysis during code generation
4. **Clarity**: Visitor focuses on code generation, not analysis
5. **Extensibility**: Easier to add new call types or behaviors

## Success Criteria

- All 379 tests continue to pass
- Python visitor code reduced by 200+ lines
- Generated Python code unchanged
- Performance improved or unchanged

## Risks and Mitigations

- **Risk**: Breaking existing functionality
- **Mitigation**: Feature flag for gradual migration

- **Risk**: Missing edge cases
- **Mitigation**: Comprehensive testing before removing old logic

- v0.66: Add type inference based on resolved calls
