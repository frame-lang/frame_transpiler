# Frame v0.46 Design - Advanced Class Features

**Created**: January 27, 2025  
**Status**: Design Document  
**Target**: Frame v0.46

## Executive Summary

Frame v0.46 extends the basic class support from v0.45 with essential OOP features including inheritance, properties, class methods, and special methods. This design document outlines the implementation approach based on Python compatibility while maintaining Frame's simplicity.

## Design Decisions

### 1. Special Methods (Dunder Methods)
**Decision**: Support both Frame names and Python dunder names
```frame
class Point {
    # Both work - developer choice
    fn init(x, y) { ... }      # Frame style
    fn __init__(x, y) { ... }   # Python style
    
    # Common special methods
    fn __str__() { ... }        # String representation
    fn __repr__() { ... }       # Debug representation
    fn __eq__(other) { ... }    # Equality comparison
    fn __lt__(other) { ... }    # Less than comparison
    fn __add__(other) { ... }   # Addition operator
    fn __sub__(other) { ... }   # Subtraction operator
    fn __mul__(other) { ... }   # Multiplication operator
    fn __len__() { ... }        # Length for collections
    fn __getitem__(key) { ... } # Indexing support
    fn __setitem__(key, val) { ... } # Index assignment
    fn __contains__(item) { ... }    # 'in' operator
    fn __iter__() { ... }       # Iteration support
    fn __next__() { ... }       # Iterator protocol
    fn __call__(*args) { ... } # Make instance callable
}
```

### 2. Class Inheritance
**Decision**: Single inheritance with `extends` keyword and `super` access
```frame
class Animal {
    var species = "Unknown"
    
    fn init(name) {
        self.name = name
    }
    
    fn speak() {
        return "Some sound"
    }
}

class Dog extends Animal {
    fn init(name, breed) {
        super.init(name)  # Call parent constructor - can be anywhere
        self.breed = breed
        self.species = "Canis familiaris"
    }
    
    fn speak() {
        return "Woof! My name is " + self.name
    }
    
    fn fetch() {
        return "Fetching!"
    }
}
```

### 3. Property Syntax
**Decision**: Python-style @property decorator with setter/deleter
```frame
class Temperature {
    fn init(celsius = 0) {
        self._celsius = celsius
    }
    
    @property
    fn celsius() {
        return self._celsius
    }
    
    @celsius.setter
    fn celsius(value) {
        if value < -273.15 {
            raise ValueError("Temperature below absolute zero")
        }
        self._celsius = value
    }
    
    @property
    fn fahrenheit() {
        return self._celsius * 9/5 + 32
    }
    
    @fahrenheit.setter
    fn fahrenheit(value) {
        self.celsius = (value - 32) * 5/9  # Uses celsius setter
    }
    
    @fahrenheit.deleter
    fn fahrenheit() {
        print("Deleting fahrenheit")
        del self._celsius
    }
}
```

### 4. Class Methods vs Static Methods
**Decision**: Keep Python's distinction between @classmethod and @staticmethod
```frame
class User {
    var user_count = 0
    
    fn init(name, email) {
        self.name = name
        self.email = email
        User.user_count = User.user_count + 1
    }
    
    @classmethod
    fn from_json(cls, json_string) {
        # cls is the class itself, can create instances
        data = parse_json(json_string)
        return cls(data["name"], data["email"])
    }
    
    @classmethod
    fn get_user_count(cls) {
        return cls.user_count
    }
    
    @staticmethod
    fn validate_email(email) {
        # No cls or self - pure utility function
        return "@" in email and "." in email
    }
}
```

### 5. Type Annotations
**Decision**: No type annotations in v0.46, will add gradual typing in v0.47+
```frame
# v0.46 - No types required
class Person {
    var name
    var age = 0
    
    fn init(name, age) {
        self.name = name
        self.age = age
    }
}

# Future v0.47+ - Optional gradual typing
class Person {
    var name: str
    var age: int = 0
    
    fn init(name: str, age: int) -> None {
        self.name = name
        self.age = age
    }
}
```

### 6. Super Constructor Flexibility
**Decision**: Allow `super.init()` anywhere in constructor (Python style)
```frame
class Dog extends Animal {
    fn init(name, breed) {
        # Can do setup before calling super
        self.breed = breed
        print("Creating a dog of breed: " + breed)
        
        # super.init() doesn't have to be first
        super.init(name)
        
        # Can do more setup after
        self.tricks = []
    }
}
```

## Implementation Plan

### Phase 1: Core Inheritance
1. **Scanner Updates**
   - Add `extends` keyword
   - Add `super` keyword
   - Add `cls` keyword (for class methods)

2. **AST Extensions**
   ```rust
   pub struct ClassNode {
       pub name: String,
       pub parent: Option<String>,  // NEW: Parent class name
       pub methods: Vec<Rc<RefCell<MethodNode>>>,
       pub class_methods: Vec<Rc<RefCell<MethodNode>>>,  // NEW
       pub properties: Vec<Rc<RefCell<PropertyNode>>>,   // NEW
       // ... existing fields
   }
   
   pub struct PropertyNode {
       pub name: String,
       pub getter: Option<Rc<RefCell<MethodNode>>>,
       pub setter: Option<Rc<RefCell<MethodNode>>>,
       pub deleter: Option<Rc<RefCell<MethodNode>>>,
   }
   ```

3. **Parser Updates**
   - Parse `extends ClassName` after class name
   - Handle `super.method()` calls
   - Track class inheritance chain

### Phase 2: Decorators
1. **@classmethod Support**
   - Parse decorator before method
   - Mark method as class method in AST
   - Generate with `cls` parameter instead of `self`

2. **@property Support**
   - Parse @property decorator
   - Parse @name.setter and @name.deleter
   - Group related methods into PropertyNode

### Phase 3: Special Methods
1. **Dual Naming Support**
   - Accept both `init` and `__init__`
   - Map Frame names to Python equivalents
   - Support all common dunder methods

2. **Operator Overloading**
   - Enable arithmetic operators via special methods
   - Enable comparison operators
   - Enable container protocols

### Phase 4: Python Visitor
1. **Inheritance Generation**
   ```python
   class Dog(Animal):  # Generate parent class
       def __init__(self, name, breed):
           super().__init__(name)  # Generate super calls
   ```

2. **Property Generation**
   ```python
   @property
   def celsius(self):
       return self._celsius
   
   @celsius.setter
   def celsius(self, value):
       self._celsius = value
   ```

3. **Class Method Generation**
   ```python
   @classmethod
   def from_json(cls, json_string):
       return cls(data["name"])
   ```

## Migration Impact

### Backward Compatibility
- All v0.45 classes continue to work unchanged
- New features are opt-in
- No breaking changes to existing code

### Forward Compatibility
- Design leaves room for future features:
  - Multiple inheritance via mixins (v0.47+)
  - Type annotations (v0.47+)
  - Access modifiers (v0.48+)
  - Abstract classes/interfaces (v0.49+)

## Success Criteria

1. ✅ Single inheritance with `extends` keyword works
2. ✅ `super.method()` calls work anywhere in methods
3. ✅ @property decorator creates properties with getters/setters
4. ✅ @classmethod creates class methods with `cls` parameter
5. ✅ Special methods enable operator overloading
6. ✅ Both Frame and Python method names work
7. ✅ All tests pass with 100% success rate
8. ✅ Generated Python code is clean and idiomatic

## Risk Mitigation

### Complexity Risk
- Implement features incrementally
- Each phase must pass tests before next phase
- Keep inheritance single (no multiple inheritance)

### Parser Ambiguity Risk
- `super` is a keyword, not a variable
- Clear precedence for decorator syntax
- Property decorators must reference existing property

### Code Generation Risk
- Test generated Python thoroughly
- Ensure proper method resolution order (MRO)
- Handle edge cases (calling super in class without parent)

Frame v0.46 will provide essential OOP features that developers expect while maintaining Frame's simplicity and Python compatibility. The design prioritizes practical features that enable real-world use cases while avoiding complexity that could confuse users or complicate the implementation.
