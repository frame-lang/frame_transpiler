# Frame v0.32 Achievements

## Release Overview
Frame v0.32 introduces comprehensive enum enhancements that bring the language's enum system to feature parity with modern programming languages. This release adds support for custom values, string enums, iteration capabilities, and module-level enum declarations.

## Major Features

### 1. Custom Enum Values
Frame now supports explicit value assignment for enum members, including negative values and auto-increment behavior.

#### Integer Enums with Custom Values
```frame
enum HttpStatus {
    Ok = 200
    Created = 201
    BadRequest = 400
    NotFound = 404
    ServerError = 500
}
```

#### Negative Values and Auto-Increment
```frame
enum Priority {
    Silent = -1     // Explicit negative value
    Low = 1         // Explicit positive value
    Medium = 5      // Explicit value
    High            // Auto: 6 (increments from previous)
    Critical        // Auto: 7
}
```

### 2. String Enums
New type annotation syntax allows creation of string-valued enums, perfect for configuration values and API constants.

```frame
enum Environment : string {
    Development = "dev"
    Staging = "staging"
    Production = "prod"
    Testing     // Auto-generated: "Testing"
}
```

### 3. Enum Iteration
Full support for iterating over enum values using `for...in` loops, with automatic type detection and property access.

```frame
// Iterate over all enum values
for status in HttpStatus {
    print(status.name + " = " + status.value)
}

// Works with string enums too
for env in Environment {
    if env == Environment.Production {
        print("Production mode: " + env.value)
    }
}
```

### 4. Module-Level Enums
Enums can now be declared at module scope, making them accessible from all functions and systems in the module.

```frame
// Module-level enum (global scope)
enum GlobalStatus {
    Inactive
    Active
    Suspended
}

fn main() {
    var status = GlobalStatus.Active
    print("Status: " + status.name)
}

system Monitor {
    interface:
        checkStatus(s: GlobalStatus)
    
    machine:
        $Ready {
            checkStatus(s: GlobalStatus) {
                if s == GlobalStatus.Active {
                    print("System is active")
                }
                return
            }
        }
}
```

### 5. Enhanced Enum Properties
All enums now provide consistent property access:
- `.name` - Returns the enum member name as a string
- `.value` - Returns the underlying value (integer or string)

## Technical Improvements

### AST Enhancements
- New `EnumType` enum distinguishing Integer vs String enums
- Flexible `EnumValue` enum supporting Integer, String, and Auto values
- Module-level enum support in `FrameModule` structure
- Enhanced `ForStmtNode` with enum iteration tracking

### Parser Improvements
- Type annotation parsing for enum declarations
- Support for negative numbers in enum values
- String literal parsing in enum declarations
- Automatic enum iteration detection in for loops
- Proper symbol table integration for enum types

### Code Generation
- Correct Python `Enum` class generation for all enum types
- Proper value assignment for custom and auto-generated values
- Enum iteration support with Python's enum iteration
- Conditional import generation (only when enums are used)

## Breaking Changes
None - All existing enum code continues to work. New features are additive.

## Migration Guide

### Upgrading Existing Enums
Existing enums continue to work without changes:
```frame
// Still valid - auto-increments from 0
enum Status {
    Pending
    Active
    Complete
}
```

### Adding Custom Values
Simply add `= value` to any enum member:
```frame
enum Status {
    Pending = 0
    Active = 1
    Complete = 100  // Jump to custom value
    Archived        // Auto: 101
}
```

### Converting to String Enums
Add `: string` type annotation and optionally provide string values:
```frame
enum Status : string {
    Pending = "PENDING"
    Active = "ACTIVE"
    Complete  // Auto: "Complete"
}
```

## Examples

### HTTP Server with Status Codes
```frame
enum HttpStatus {
    Ok = 200
    Created = 201
    BadRequest = 400
    NotFound = 404
    ServerError = 500
}

system HttpServer {
    interface:
        handleRequest() : HttpStatus
    
    machine:
        $Ready {
            handleRequest() : HttpStatus {
                // Process request...
                if error {
                    return HttpStatus.ServerError
                }
                return HttpStatus.Ok
            }
        }
}
```

### Configuration with String Enums
```frame
enum LogLevel : string {
    Debug = "DEBUG"
    Info = "INFO"
    Warning = "WARN"
    Error = "ERROR"
}

fn configureLogging(level: LogLevel) {
    print("Setting log level to: " + level.value)
    
    // Show all available levels
    for l in LogLevel {
        print("Available: " + l.name + " (" + l.value + ")")
    }
}
```

### Priority System with Mixed Values
```frame
enum Priority {
    None = 0
    Low = 10
    Medium = 50
    High = 100
    Critical = 1000
}

system TaskManager {
    interface:
        addTask(name: string, priority: Priority)
    
    machine:
        $Managing {
            addTask(name: string, priority: Priority) {
                print("Adding task: " + name)
                print("Priority level: " + priority.value)
                
                if priority.value >= Priority.High.value {
                    print("HIGH PRIORITY TASK!")
                }
                
                return
            }
        }
}
```

## Bug Fixes

### Enum Qualification Bug Fix
Fixed a critical bug where enum member references in generated Python code were not properly qualified with system names, causing `NameError` at runtime. The fix ensures:
- System-scoped enums are qualified with system name (e.g., `HttpServer_HttpStatus.Ok`)
- Module-level enums remain unqualified
- Proper handling of dot-notation enum references in the Python visitor

**Impact**: All 6 failing enum tests now pass, achieving 100% test success rate (170/170).
