# Frame v0.43 - Python Type Annotations Support

## Overview
Frame v0.43 introduces comprehensive support for Python type hints/annotations, enabling better code documentation, IDE support, and optional static type checking. The implementation maintains Frame's clean syntax while generating proper Python type annotations.

## Key Features

### 1. Function Parameter Type Annotations ✅
```frame
fn add(a: int, b: int) : int {
    return a + b
}
```
Generates:
```python
def add(a: int, b: int) -> int:
    return a + b
```

### 2. Function Return Type Annotations ✅
- **Syntax**: `: type` after parameter list
- **All Python types supported**: int, float, str, bool, list, dict, set, tuple, any
- **Works with async functions**: `async fn fetch(url: str) : str`

### 3. Variable Type Annotations ✅
```frame
fn example() {
    var count: int = 0
    var name: str = "Frame"
    var items: list = [1, 2, 3]
    var config: dict = {"key": "value"}
}
```
Generates:
```python
def example():
    count: int = 0
    name: str = "Frame"
    items: list = [1, 2, 3]
    config: dict = {"key": "value"}
```

### 4. Supported Types

#### Basic Types
- `int` - Integer numbers
- `float` - Floating-point numbers
- `bool` - Boolean values (true/false)
- `string` or `str` - Text strings (both styles supported)

#### Collection Types
- `list` - Ordered mutable sequences
- `dict` - Key-value mappings
- `set` - Unordered unique elements
- `tuple` - Ordered immutable sequences

#### Special Types
- `any` - Any type (dynamic typing)
- User-defined types pass through as-is

## Type System Design

### Frame to Python Type Mapping
| Frame Type | Python Type | Notes |
|------------|-------------|-------|
| `int` | `int` | Integer values |
| `float` | `float` | Floating-point values |
| `bool` | `bool` | Boolean values |
| `string` | `str` | Frame's string type |
| `str` | `str` | Python-style alias |
| `list` | `list` | Basic list type |
| `dict` | `dict` | Dictionary type |
| `set` | `set` | Set type |
| `tuple` | `tuple` | Tuple type |
| `any` | `any` | Any type |

### Implementation Details

#### Parser Support
- Types are parsed using `: type` syntax after identifiers
- Return types use `: type` after parameter list
- Type information stored in AST nodes (TypeNode)

#### AST Integration
- `FunctionNode` has `type_opt: Option<TypeNode>` for return types
- `ParameterNode` has `param_type_opt: Option<TypeNode>` for parameter types
- `VariableDeclNode` has `type_opt: Option<TypeNode>` for variable types

#### Python Visitor
- `format_type()` method converts Frame types to Python equivalents
- Type annotations generated inline with declarations
- Empty type strings result in no annotation (backward compatible)

### Mixed Typing (Gradual Typing)
Frame supports gradual typing - you can mix typed and untyped code:
```frame
fn mixed_example(typed_param: int, untyped_param) {
    var typed_var: str = "explicit"
    var untyped_var = "inferred"
    return typed_param + untyped_param
}
```

## Migration Guide

### Adding Types to Existing Code
1. **Start with function signatures**: Add parameter and return types
2. **Add variable types selectively**: Focus on public APIs and complex data
3. **Use gradual typing**: Not everything needs types immediately

### Type Syntax Quick Reference
```frame
# Parameters
fn func(param: type) { }

# Return types
fn func() : type { }

# Variables
var name: type = value

# Combined
fn process(data: list, flag: bool) : dict {
    var result: dict = {}
    return result
}
```

## Limitations

### Current Limitations
1. **No generic types yet**: `list[int]` syntax not supported (planned for future)
2. **No union types**: `int | str` not supported yet
3. **No type aliases**: Custom type definitions not yet implemented
4. **No Optional types**: `Optional[int]` not supported yet

### Future Enhancements
- Generic type parameters: `list[T]`, `dict[K, V]`
- Union types: `int | str | None`
- Type aliases: `type UserId = int`
- Optional types: `Optional[T]`
- Protocol types for structural typing

Frame v0.43 brings Python type annotations to Frame, enabling better tooling support and code clarity while maintaining Frame's simple syntax. The implementation is complete for basic types and provides a solid foundation for future enhancements like generics and union types.
