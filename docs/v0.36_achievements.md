# Frame v0.36 Achievements - Event-Handlers-as-Functions Architecture

## Overview
Version 0.36 introduces a fundamental architectural improvement in how Frame generates event handlers for state machines. Instead of generating monolithic state methods containing all event handlers, v0.36 generates each event handler as an individual function, with state methods serving as lightweight dispatchers.

## Key Achievements

### 1. Event-Handlers-as-Functions Architecture ✅
- **Individual Handler Functions**: Each event handler is generated as a separate function
- **State Dispatchers**: State methods become simple routers that dispatch to appropriate handlers
- **Cleaner Code Generation**: More modular and maintainable generated code
- **Better Separation of Concerns**: Each handler is self-contained with its own logic

### 2. Special Event Name Handling ✅
- **Enter Event**: `$>` events generate functions named `__handle_{state}_enter`
- **Exit Event**: `<$` events generate functions named `__handle_{state}_exit`
- **Valid Python Identifiers**: Automatic conversion ensures all function names are valid Python
- **Consistent Naming**: Both handler generation and dispatcher use same naming logic

### 3. Configuration System ✅
- **Feature Flag**: `event_handlers_as_functions` flag in PythonConfig
- **Backward Compatible**: Flag allows gradual migration without breaking existing code
- **Default Behavior**: When disabled, generates traditional monolithic state methods

### 4. Async Foundation ✅
- **Async Detection**: Individual handlers detect if they contain await expressions
- **Automatic async def**: Handlers with await are generated as `async def` functions
- **State Dispatcher Async**: Dispatchers become async when they have async handlers
- **Proper await Propagation**: Async dispatchers properly await async handlers

### 5. Test Results
- **Success Rate**: 208/209 tests passing (99.5% success rate)
- **Non-Async Tests**: All 208 non-async tests passing
- **Remaining Work**: 1 async test requires full async runtime support

## Technical Implementation

### Generated Code Structure (Before v0.36)
```python
def __state_Ready(self, __e):
    if __e._message == "start":
        # Inline handler code for start
        print("Starting")
        self.__transition(Working)
    elif __e._message == "stop":
        # Inline handler code for stop
        print("Stopping")
        return
```

### Generated Code Structure (After v0.36)
```python
def __handle_ready_start(self, __e, compartment):
    print("Starting")
    self.__transition(Working)
    return

def __handle_ready_stop(self, __e, compartment):
    print("Stopping")
    return

def __state_Ready(self, __e, compartment):
    if __e._message == "start":
        return self.__handle_ready_start(__e, compartment)
    elif __e._message == "stop":
        return self.__handle_ready_stop(__e, compartment)
```

### Async Handler Example
```python
async def __handle_ready_getData(self, __e, compartment):
    data = await fetch_data(__e._parameters["id"])
    self.return_stack[-1] = data
    return

async def __state_Ready(self, __e, compartment):
    if __e._message == "getData":
        return await self.__handle_ready_getData(__e, compartment)
```

## Benefits

### 1. **Improved Code Organization**
- Each handler is a distinct function with clear boundaries
- Easier to understand and debug individual event handlers
- Better IDE support (jump to definition, code folding)

- Individual handlers can be async or sync as needed
- Proper async detection and generation per handler
- Sets stage for full async runtime support

### 3. **Better Testing**
- Individual handlers can be tested in isolation
- Clearer stack traces when errors occur
- Easier to mock and stub individual handlers

### 4. **Performance Potential**

- Potential for handler caching or compilation
- Better JIT optimization opportunities
