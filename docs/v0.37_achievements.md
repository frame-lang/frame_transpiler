# Frame v0.37 Achievements

## Overview
Frame v0.37 completes the async/await implementation and documents known limitations with index operations.

## Test Success Rate: 99.5% (211/212 tests passing)

## Key Achievements

### 1. Complete Async/Await Support ✅
- Full async function declarations (`async fn`)
- Async interface methods in systems
- Await expressions throughout the language
- Mixed sync/async systems with proper runtime handling
- Individual async event handlers with correct code generation

### 2. Mixed Sync/Async Systems ✅
Systems can now have both synchronous and asynchronous interface methods:
```frame
system MixedSystem {
    interface:
        getData()           // Sync method
        async fetchData()   // Async method
        processSync()       // Sync method
        async processAsync() // Async method
}
```

### 3. Async Runtime Handling ✅
- `__kernel_sync` wrapper for sync interface methods calling async kernel
- Proper thread pool executor for sync contexts
- Clean async/await propagation through event handlers

### 4. Documentation Updates ✅
- Grammar documentation updated to v0.37
- Index operation limitations clearly documented
- Workarounds provided for affected users
- Dev notes updated with known issues

## Known Limitations

### Index Operations
Frame's parser does not fully support native index operations. This affects:
- Dictionary access: `dict[key]`
- List access: `array[index]`
- Assignment to indexed items: `self.results[key] = value`

**Workaround**: Use backtick expressions for index operations:
```frame
// Instead of: self.results[str(id)] = value
// Use: self.results`[str(id)]` = value
```

## Technical Details

### Parser Issue
The Frame parser incorrectly interprets `expression[index]` as two separate statements:
1. A call chain expression: `expression`
2. A literal/SuperString: `[index]`

This causes the generated code to have incorrect line breaks:
```python
# Generated (incorrect):
self.results
[str(id)] = value

# Should be:
self.results[str(id)] = value
```

### Visitor Improvements
Added detection in `visit_literal_expression_node` to identify index patterns and avoid adding newlines, but this only helps when backticks are used.

## Future Work

### Proper Index Operation Support
To fully resolve this issue would require:
1. Adding index operation support to the grammar
2. Creating an IndexExprNode AST node type
3. Updating the parser to recognize index syntax
4. Modifying all visitors to handle index operations

This is a significant change that would affect the core language grammar.

## Migration Guide

For users affected by index operation issues:

1. **Dictionary/List Access**: 
   ```frame
   // Old (problematic):
   var item = mylist[0]
   
   // New (working):
   var item = mylist`[0]`
   ```

2. **Dictionary Assignment**:
   ```frame
   // Old (problematic):
   self.data[key] = value
   
   // New (working):
   self.data`[key]` = value
   ```

3. **Complex Operations**:
   ```frame
   // Use full backtick blocks:
   `
   self.results[str(task_id)] = value
   item = my_list[index]
   my_dict[key] = new_value
   `
   ```
