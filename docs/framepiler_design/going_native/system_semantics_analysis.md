# Going Native: System Semantics Analysis (Thorough)

Status: Draft for going_native branch
Last updated: 2025‑11‑05

This document defines precise, language‑agnostic semantics for Frame “system” blocks while accommodating native language syntax inside member bodies. It enumerates tricky cases, cross‑language differences, runtime obligations, and open design choices that affect correctness, diagnostics, and developer experience.

## 1) System Anatomy and Normalization

A Frame system is normalized internally to:
- Metadata: system name, target language, module path.
- Blocks:
  - interface: callable methods visible to host code, each with a default return value expression.
  - operations: pure or impure helpers (callable from actions/handlers).
  - actions: side‑effecting helpers; may call operations.
  - domain: stateful properties and enums (system‑scoped variables and constants).
  - machine: hierarchical state machine (HSM) with states, event handlers, enter/exit handlers.

Normalization invariants to keep across targets:
- Run‑to‑completion handler semantics (no concurrent execution of two handlers in the same system instance).
- Deterministic enter/exit order for transitions.
- Deterministic effect of state stack push/pop semantics.
- Deterministic mapping of interface method invocations to events (and return values via return_stack or equivalent).

## 2) Event/Message Model and Dispatch

2.1 Event representation
- Minimum: message string + parameter map (string→any) + optional return slot.
- Native expansions: typed payloads (TS), dataclasses (Py), etc. remain internal to bodies.

2.2 Dispatch sources
- Interface invocations: translated into FrameEvent("methodName", params) and passed to kernel.
- Internal events: may be generated by actions/operations/handlers (e.g., forward)
- External sources: sockets, timers, IO callbacks (as supported by runtime).

2.3 Dispatch model
- Single‑threaded logical model per instance; ensure run‑to‑completion: a handler runs to completion (transition or fall‑through return) before the next event begins.
- Reentrancy policy:
  - Synchronous: actions/operations may call the kernel (forward/transition) inside a handler. This is allowed but must set next_compartment and return immediately to maintain RTC semantics.
  - Asynchronous: in Py/TS, `await`/Promise microtasks are allowed only in async handlers; we ensure transitions finalize before resuming awaiting code unless the handler itself is async and yields explicitly.

2.4 Event loop binding (language‑specific)
- Python: asyncio event loop (optional). If any handler uses `await` or system is marked async, kernel operations that schedule follow‑up events should cooperate with the loop (e.g., use `asyncio.create_task`) or remain synchronous if the handler uses no `await`.
- TS/Node: Promise microtask queue. Kernel dispatch remains synchronous until MIR glue needs to return control, then subsequent steps may be queued.

## 3) Transition Semantics

3.1 External vs Internal transitions
- External transition (default): exit current state (run exit), change state, run enter of target; event terminates.
- Internal transition (optional future): remain in state, no exit/enter; not modeled yet—if added, must be explicit.

3.2 Enter/Exit ordering
- exit(current) → (optional push/pop changes) → set nextCompartment → enter(target). If there is a queued forward event, deliver it after enter completes, then clear.

3.3 Transitions from within actions/operations
- Allowed; MIR glue sets `next_compartment` and emits `return` immediately. Code after transition is unreachable (warn).

3.4 Transition arguments
- `-> $State(expr1, expr2, ...)` are evaluated as native expressions in the current scope and set as enter arguments of the target state. In Py: `self.__compartment.enter_args = {...}`; in TS: `this.__compartment.enterArgs`.

3.5 Unreachable code
- After a transition in a handler/action/operation, any subsequent native code is unreachable in the same logical event. We should emit warnings and optionally offer a linter rule.

## 4) Parent Forward (=> $^)

4.1 Semantics
- Deliver the current event to the parent state handler immediately. No implicit exit/enter; if a transition happens in parent, that governs.

4.2 Runtime glue
- Py: `self._frame_router(__e, compartment.parent_compartment)` if available, followed by `return`.
- TS: `this._frame_router(__e, compartment.parentCompartment)` and `return`.

4.3 Edge cases
- No parent: warn or no‑op (design choice); default to no‑op + warning.
- Forward followed by code: unreachable (warn).

## 5) State Stack (Push/Pop)

5.1 Semantics
- `$$[+]`: snapshot current state/compartment onto a stack; continues execution.
- `$$[-]`: pop the previous snapshot; transition to the popped state; return immediately (unreachable code afterward).

5.2 Runtime glue
- Py: maintain `self.return_stack` (for returns) and a separate `state_stack` (if not present, add); glue sets next compartment and returns.
- TS: same pattern with `this.returnStack`, `this.stateStack`.

5.3 Edge cases
- Empty pop: error or no‑op? Recommend error (compile‑time if provable; otherwise runtime exception). Provide diagnostic.

## 6) Interface Methods & Returns

6.1 Mapping
- Interface method call → construct event E with params → dispatch → return last return_stack value.
- Default return in interface declaration used if handler never sets the return_stack slot.

6.2 Async interface
- Language runtimes may allow async interface wrappers that return Promises/futures. For TS/async Py, generate async wrappers when any handler in the target path is async.

6.3 Edge cases
- Multiple returns: last write to return slot wins before exiting handler.
- Nested calls: allowed; still obey RTC semantics due to immediate returns on transitions.

## 7) Operations vs Actions

7.1 Semantics
- Actions: side‑effecting helpers intended to be called from handlers/operations; may contain transitions/forward/stack directives.
- Operations: logically pure (recommended), but we won’t enforce purity; may call actions; transitions here are allowed but discouraged (warn).

7.2 MixedBody policy
- Both are MixedBody. Directive presence is allowed and expands via MIR.

## 8) Domain Block & Enums

8.1 Domain variables
- Initialized at system construction; updates from native code must be deterministic; maintain per‑instance state.

8.2 Enums
- Provide target‑native enum constructs (Py Enum/TS enum/Java enum) emitted where referenced. Keep them in codegen as needed.

## 9) Error Semantics & Exceptions

9.1 Native exceptions
- If a native exception is thrown inside a handler and not caught, treat as an execution failure for that event (test runner fails). Optionally convert to a special `$Error` event in future versions.

9.2 Transition glue errors
- Glue templates must not throw unless state/stack invariants are violated. Provide meaningful messages (e.g., “pop on empty stack”).

9.3 Diagnostics
- Dual mapping for errors: frame> line N, target> line M; include target snippet.

## 10) Concurrency & Reentrancy

10.1 Single‑threaded model per instance
- Handlers run fully before the next event logically begins. Native async constructs yield only when permitted by async handlers; transitions conclude the logical handler.

10.2 Reentrancy controls
- Disallow reentrant dispatch except via explicit forward/transition sequences. Avoid deadlocks with reentrant calls by ensuring transitions return immediately.

10.3 Timers/IO
- Future runtime APIs should queue events to the kernel; do not invoke handlers recursively directly.

## 11) Outline & Boundary Detection

11.1 OutlineScanner responsibilities
- Find systems, sections, and member bodies precisely.
- For Python: track indentation; body ends at dedent of the section/def block.
- For TS: track brace depth; body ends at matching closing brace at depth==0.

11.2 Body slicing correctness
- Native segment spans must exclude section and member closing tokens (e.g., `}` lines) and any following structural lines.

11.3 Edge cases
- Lines with trailing braces/comments; triple‑quoted strings; template literals; nested braces in strings/macros; preprocessor lines in C/C++.

## 12) MixedBody Details

12.1 Items
- NativeAst: derived from SWC/rustpython (later: tree‑sitter for other languages).
- NativeText: fallback when AST not available or comments need preservation.
- FrameStmt: MIR { Transition, ParentForward, StackPush, StackPop, Return }

12.2 Ordering
- Emit in the order found; MIR expansions insert return/early exit as required.

12.3 Unreachable warnings
- Post‑processing scan per body to annotate unreachable native code following MIR operations that imply termination for the current handler.

## 13) Multi‑file & Linking

13.1 Consolidated imports
- For TS/Java: linker emits a single shared runtime import and filters per‑module duplicates to avoid redeclarations.

13.2 Output directory discipline
- All generated modules and runtime reside in a single “generated” outDir so relative imports resolve.

13.3 Python
- Keep single flat module; multi‑file mode is optional (package) and uses `__init__.py` if requested.

## 14) Source Maps & Tooling

14.1 Source maps
- Py: JSON mapping with frameLine→pyLine spans per MixedBody.
- TS: v3 sourcemaps if using SWC codegen; compose expansions.

14.2 Debug output
- --debug-output prints generated code + structured mapping for IDE tooling.

## 15) Cross‑Language Tricky Scenarios

15.1 Transitions in expression contexts
- Example: inside a Python list comprehension or TS arrow body. Policy: allow only at SOL; warn/error if found in expression contexts where control‑flow is ambiguous.

15.2 Captured locals after transition
- Closures in Py/TS that capture locals read after a transition are undefined. Warn if a transition appears before such closures.

15.3 Shadowing domain variables
- Prohibit redeclaration of domain names inside handlers/actions for clarity (existing rule).

15.4 Keyword/operator collisions
- TS `=>` vs Frame `=> $^` addressed by requiring `$^` and SOL.
- C++ `->` pointer deref vs Frame `-> $`.

15.5 Preprocessor/macros (C/C++)
- Do not segment Frame directives inside macro bodies; SOL detection under macro expansions should be disabled.

## 16) Validation Matrix (TCK‑style)

- Handler lifecycle: run‑to‑completion, transitions, returns, forward, push/pop
- Interface returns (default vs set by handler)
- Nested transitions/forwards
- MixedBody with multiple segments and directives
- Async handler behavior (Py/TS)
- Multi‑file TS linking (shared runtime import)
- Error propagation from native exceptions
- Boundary detection (Python indentation, TS braces, templates, f‑strings)

## 17) Open Questions (Decision Requests)

1) Enforce MixedBody as the only handler body path (drop legacy per‑token parsing in handlers)?
2) Allow target‑native signatures in Frame headers now, or keep them only inside native bodies for the near term?
3) Reentrancy policy strictness: disallow any nested kernel dispatch except via MIR (forward/transition), or allow a controlled subset?
4) Errors after transition: warnings only or compile‑time errors?
5) Macro handling in C/C++: segmentation only vs early tree‑sitter integration?
6) Standardize source maps for all targets (TS sourcemap v3 + Python JSON)?
7) Runtime import ownership: linker‑owned by default (recommended) vs per‑module opt‑in.
