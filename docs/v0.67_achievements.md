## Overview

Frame v0.67 fixes a critical source map bug where state handler content was mapped to incorrect Python lines, causing debugging confusion. This one-line fix ensures accurate source-to-target line mappings for all event handlers.

## Bug Fixed

### Source Map Off-by-One for State Handler Content ✅

**Issue**: The transpiler's source map was mapping Frame lines to the wrong Python lines for state handler content. Specifically, Frame statements inside event handlers were being mapped to the function definition line instead of their actual corresponding lines.

**Example of the Problem**:
```frame
$Running {
    $>() {                              // Line 28
        print("FirstSystem running")     // Line 29
        return                          // Line 30
    }
}
```

**Before v0.67** (incorrect mapping):
- Frame line 29 → Python line 65 (function definition) ❌
- Frame line 30 → Python line 66 (print statement) ❌

**After v0.67** (correct mapping):
- Frame line 28 → Python line 64 (function definition) ✅
- Frame line 29 → Python line 65 (print statement) ✅
- Frame line 30 → Python line 66 (return statement) ✅

## Technical Implementation

### Root Cause
The `generate_event_handler_function()` method was generating the Python function definition without adding a source mapping for the event handler's declaration line. This caused all subsequent statements to be off by one line.

### The Fix
Added a single line to map the event handler declaration to its Python function definition:

```rust
// In generate_event_handler_function() at line 475
self.add_source_mapping(evt_handler_node.line);
```

- `framec/src/frame_c/visitors/python_visitor.rs` - Added source mapping for event handler function definitions
