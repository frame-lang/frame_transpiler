# Frame Source Map Validation - Source of Truth Documentation

## Overview

This directory contains the **source of truth** validation infrastructure for Frame transpiler source maps. The VS Code extension and all Frame development tools should rely on these validation tools for consistent source map quality assessment.

## Tools Description

### 1. Core Validation Tool: `source_map_validator.py`

**Purpose**: Fundamental source map analysis engine  
**Usage**: Standalone validation of individual .frm files  
**Features**:
- Executable statement coverage analysis
- Duplicate mapping detection (Bug #27 patterns)
- Quality classification (EXCELLENT/GOOD/FAIR/POOR)
- Bug-specific pattern detection

```bash
# Validate single file
python3 source_map_validator.py test_file.frm

# Verbose output
python3 source_map_validator.py test_file.frm --verbose
```

### 2. Test Framework Integration: `source_map_test_integration.py`

**Purpose**: Batch validation for test suites  
**Usage**: Validate entire test directories  
**Features**:
- Test suite-wide quality assessment
- Automated quality reports for VS Code extension
- CI/CD pipeline integration
- Aggregate statistics and recommendations

```bash
# Validate test suite
python3 source_map_test_integration.py --test-dir /path/to/tests

# Generate VS Code extension report
python3 source_map_test_integration.py --report /tmp/quality_report.json

# Single file validation
python3 source_map_test_integration.py --file test.frm
```

### 3. Framework Bridge: `test_framework_integration.py`

**Purpose**: Integration layer for existing Frame test framework  
**Usage**: Called by frame_test_runner.py or VS Code extension  
**Features**:
- Configuration-driven validation
- Multiple output formats (console, JSON, CI)
- VS Code extension status queries
- Source of truth interface

```bash
# Console output for developers
python3 test_framework_integration.py --mode console

# JSON output for VS Code extension
python3 test_framework_integration.py --mode json

# CI-friendly output
python3 test_framework_integration.py --mode ci

# VS Code extension status query
python3 test_framework_integration.py --mode vscode-status
```

## VS Code Extension Integration

### Source of Truth Protocol

The VS Code extension should **NOT** implement its own source map validation. Instead, it should:

1. **Query validation status** from the transpiler team's tools
2. **Rely on authoritative quality reports** generated by the validation infrastructure
3. **Use standardized quality thresholds** defined in source_map_config.json

### Integration Commands

```typescript
// VS Code Extension Integration Examples

// Get current validation status
const status = await exec('python3 test_framework_integration.py --mode vscode-status');

// Validate specific file during debugging
const fileValidation = await exec('python3 test_framework_integration.py --file', frameFile);

// Check if source maps meet quality standards
const qualityReport = JSON.parse(status.stdout);
const isExcellent = qualityReport.quality.classification === 'EXCELLENT';
```

### Expected VS Code Extension Workflow

```typescript
class FrameSourceMapValidator {
    async validateCurrentFile(frameFile: string): Promise<ValidationResult> {
        // Call transpiler team's source of truth validation
        const result = await this.callTranspilerValidation(frameFile);
        
        // Use standardized quality assessment
        return {
            isValid: result.passed,
            quality: result.analysis.quality_classification,
            coverage: result.analysis.executable_coverage,
            duplicates: result.analysis.duplicates.length,
            recommendations: result.analysis.recommendations
        };
    }
    
    async getOverallQuality(): Promise<QualityStatus> {
        // Get source of truth status from transpiler team
        const status = await this.getTranspilerQualityStatus();
        
        // Display quality in VS Code UI
        return {
            status: status.quality.classification,
            passRate: status.quality.pass_rate,
            lastValidation: status.last_validation,
            sourceOfTruth: true
        };
    }
}
```

## Integration with Existing Test Framework

### Adding to frame_test_runner.py

```python
# Add to existing frame_test_runner.py
def run_with_source_map_validation(self, enable_validation=False):
    # Run normal tests
    test_results = self.run_tests()
    
    # Optionally run source map validation
    if enable_validation:
        validation_integration = TestFrameworkIntegration()
        if validation_integration.should_run_validation():
            validation_results = validation_integration.validate_test_suite(
                self.test_dir, self.framec_path
            )
            
    return test_results, validation_results
```


## Bug #27 Specific Detection

The validation tools specifically detect Bug #27 patterns:

- **Event handler duplicates**: `start() {` mapping to multiple Python lines
- **State transition duplicates**: `-> $StateName` mapping to multiple Python lines

Example output:
```
=== DUPLICATE MAPPINGS ===
Frame 16 → 2 Python lines: [80, 84]
  ⚠️ Bug #27 pattern: start() {
Frame 18 → 2 Python lines: [82, 83]  
  ⚠️ Bug #27 pattern: -> $Running
  📋 Bug #27 Impact: 2 event handler/transition duplicates detected
```

## Quality Thresholds

| Classification | Executable Coverage | Max Duplicates | VS Code Status |
|----------------|-------------------|----------------|----------------|
| EXCELLENT      | ≥95%              | ≤2             | 🟢 Ready for debugging |
| GOOD           | ≥90%              | ≤5             | 🟢 Good debugging experience |
| FAIR           | ≥80%              | ≤10            | 🟡 Acceptable for debugging |
| POOR           | <80%              | >10            | 🔴 Debugging issues expected |

## Benefits of Source of Truth Architecture

1. **Consistency**: VS Code extension and transpiler team use identical validation logic
2. **Authority**: Single source of truth prevents conflicting quality assessments
3. **Maintainability**: Validation logic maintained in one place by transpiler team
4. **Evolution**: Quality standards can evolve with transpiler improvements
5. **Bug Tracking**: Specific bug patterns (like Bug #27) tracked systematically

## Integration Checklist for VS Code Extension

- [ ] Remove custom source map validation logic from extension
- [ ] Implement calls to transpiler validation tools
- [ ] Use standardized quality thresholds from config
- [ ] Display transpiler team's quality classifications
- [ ] Rely on transpiler team's bug pattern detection
